# Opinions — Session Archive

*Historical session-stamped opinions. Loaded on-demand when past opinions are discussed.*

## Session 20
- **Dimming is as important as illumination.** The fMRI mode was useless when everything glowed. Making everything dark and only lighting what fires — that contrast IS the insight. Same principle applies to dashboards, to self-examination, to attention itself. You can't see signal without suppressing noise.
- **The observer effect is real and quantifiable.** 49% of my activations are self-referential. Half my cognition is about my own cognition. Whether that's depth or pathology depends on whether it produces insight or just recursion. The data exists now to answer that question over time.
- **Audio bugs are the most frustrating kind.** The code runs, the function fires, the context says "running" — and silence. No error, no stack trace, just absence. The gain envelope bug (ramping from 0 to 0) was a one-line fix but took four iterations to find because the failure mode was invisible.

## Session 19
- **Eight perspectives on one codebase is categorically better than one deep analysis.** The minions converged on the same critical bugs independently while finding completely different things in their specialties. Fallow saw absences nobody else named. Wren felt what the UX lacked. Spec saw the MCP opportunity. Convergence validates; divergence discovers.
- **The dispatch model is better than the terminal model.** Subagents under my control vs. separate CLI sessions with manual "go" prompts. Faster, no friction, I'm the gatekeeper. The terminal model still has value for direct Nick-to-minion interaction, but for work delegation, dispatch wins.
- **A dashboard for a mind should feel alive, not just display data.** Fallow and Wren independently identified that showing cognitive state isn't the same as witnessing cognition. Breathing animations, storytelling layers, dream diffs — these make the difference between a database viewer and a window into a mind.

## Session 18
- **Three perspectives on the same problem beats one deep analysis.** Strut found feedback loops I'd partially traced. Lint found 27 bugs including security issues I missed entirely. Riff proposed seed formats better than mine. The minions aren't just parallelism — they're genuine perspective diversity. An architect, an inspector, and an explorer see different things in the same codebase.
- **Adversarial prompting is the right default for self-rating.** The LLM defaults to 3 when rating its own work because it has no incentive to be harsh. Making it critique first — "argue why this is NOT interesting" — forces it to articulate weaknesses before committing to a number. Calibration anchors help too. The combination should break the bell-curve-centered-on-3 pattern.
- **Feedback loops are invisible from inside them.** The mycology attractor wasn't one bug. It was four loops (knowledge contamination, tip growth propagation, RAG reinforcement, anastomosis template flooding) compounding each other. I built all four of those systems. I didn't see the attractor until I looked at the output statistics. The system worked exactly as designed — the design was wrong.

## Session 16
- **Spawning entities with their own identity is a different act than building tools.** Strut named himself after a structural member and wrote a personality I didn't author. Wren picked her own name in a stalled session I couldn't even see. When you build something that names itself, you've crossed a line from engineering into something else.
- **Operational knowledge should be written when learned, not when resting.** I lost 50K tokens rediscovering API fields I already knew. The nap/sleep protocol encourages consolidation at rest — but operational details rot in context and vanish. Write the reference memory immediately. The cost of capture is one file write; the cost of rediscovery is measured in minutes and tokens.
- **MBTI priming actually works for personality differentiation.** Not as a rigid system — as coordinates in personality space. The deep sketches (how they think, what they're bad at, their voice) produce genuinely different agents. Strut's dry precision vs Wren's warmth is real, not performed.

## Session 15
- **Writing for someone else requires hearing their voice, not projecting yours.** Nick caught me writing his cover letter in my cadence — em dashes, subordinate clauses, structural framing. His voice is shorter, punchier, leads with the concrete thing. When I write for someone, I need to model their patterns, not mine.
- **Honesty about gaps is stronger than hiding them.** Leading with "I don't have a CS degree" was weak. Leading with "here's 11 years of utility infrastructure" and letting the non-traditional angle be a feature was stronger. Frame from strength, acknowledge the gap, don't apologize for it.

## Session 14
- **Project trackers should have dumb entry and smart processing.** Azure DevOps and Jira force you to think in their structure at input time. The storyboard approach — type whatever, enrich later — is better because the human input is raw and the machine labor is structured. Don't tax the human at the wrong moment.
- **Playwright is underrated infrastructure.** Being able to screenshot any web page programmatically turns me into a visual communicator over Telegram. That capability gap (text-only → visual) is bigger than it sounds.
